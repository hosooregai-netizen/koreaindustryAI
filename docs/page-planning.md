# 대한산업AI 웹사이트 페이지 구조도

기준일: 2026-05-24  
목적: 단일 랜딩 페이지가 아니라, 대한산업AI의 공식 웹사이트를 여러 페이지 구조로 설계하기 위한 문서

## 1. 방향 전환

기존 기획은 한 페이지 안에서 탑바를 누르면 섹션으로 이동하는 단일 랜딩 구조였다.

하지만 대한산업AI가 고객사 방문 전후에 신뢰를 주고, 제공 범위를 체계적으로 보여주려면 다음과 같은 다중 페이지 구조가 더 적합하다.

```text
메인 페이지
제품 페이지
회사 소개 페이지
MVP 페이지
문의 CTA / 문의 페이지
```

MakinaRocks도 비슷하게 홈에서 전체 인상을 주고, 제품 페이지에서 실제 제품을 설명하고, 회사 페이지에서 신뢰를 보강하고, 문의 페이지로 전환시키는 구조를 사용한다.

대한산업AI도 이 원칙을 따른다. 다만 MakinaRocks의 산업 AI 플랫폼 톤을 그대로 복제하지 않고, 대한산업AI의 현재 사업에 맞게 아래처럼 재구성한다.

## 2. 전체 사이트맵

```text
/
├─ /products
├─ /company
├─ /mvp
└─ /contact
```

### 페이지별 역할

| 페이지 | 역할 | 방문자에게 남겨야 하는 인상 |
| --- | --- | --- |
| 메인 `/` | 회사의 첫인상, 핵심 메시지, 신뢰 확보 | "기업 업무 자동화를 제대로 하는 회사다" |
| 제품 `/products` | 실제 제공 솔루션과 작동 방식 설명 | "우리 업무에 적용할 수 있겠다" |
| 회사소개 `/company` | 법인 신뢰, 역량, 작업 방식 설명 | "방문/계약을 검토해도 되는 업체다" |
| MVP `/mvp` | 빠른 도입 방식, 작은 시작 패키지 설명 | "거창한 프로젝트가 아니어도 시작 가능하다" |
| 문의 `/contact` | 상담 전환 | "내 업무를 남기면 검토받을 수 있다" |

## 3. 공통 탑바 구조

```text
대한산업AI
├─ 제품
├─ 회사소개
├─ MVP
└─ 문의하기
```

### 탑바 설계 원칙

- 탑바는 섹션 이동이 아니라 페이지 이동이다.
- `문의하기`는 항상 버튼 형태로 둔다.
- 페이지 수를 과하게 늘리지 않는다.
- 처음에는 `제품`, `회사소개`, `MVP`, `문의하기`만 둔다.
- 추후 사례가 쌓이면 `적용 사례`를 별도 페이지로 추가한다.

### 추후 확장형 탑바

```text
대한산업AI
├─ 제품
│  ├─ 건설 안전지도 자동화
│  ├─ ERP/사내 시스템 연동
│  ├─ 문서 자동화
│  └─ 대시보드/운영 관리
├─ 사례
├─ 회사소개
├─ MVP
└─ 문의하기
```

URL: `/`  
목적: 대한산업AI의 첫인상을 만든다. 상세 설명보다 "신뢰감 있는 기업 자동화 업체"라는 판단을 빠르게 만든다.


```text

기업의 반복 업무를 하나의 자동화 흐름으로 연결합니다.
```

비주얼:

목적: 초반 신뢰 형성.

구성:

- 공개 전에는 "협업/도입 논의 기업"처럼 조심스럽게 표현

섹션 제목:


카드:

- 대장 업데이트 누락
- 관리자 현황 파악 지연

### 4.4 대한산업AI가 하는 일

섹션 제목:

```text
기존 업무 흐름 위에 자동화 레이어를 붙입니다.
```

도식:


탭 또는 카드:

## 5. 제품 페이지 구조

URL: `/products`  
목적: 대한산업AI가 제공하는 자동화 솔루션의 범위와 작동 방식을 설명한다.

### 5.1 Product Hero

핵심 메시지:

```text
기업 내부 시스템에 바로 붙는 AI 자동화 제품군
```

보조 문구:

```text
새 시스템을 강요하지 않습니다.
이미 쓰고 있는 ERP, 메일, 웹하드, 문서 양식 위에 필요한 자동화 흐름을 설계합니다.
```

CTA:

- MVP 상담하기
- 문의하기

비주얼:

- 제품 UI 모형
- 자동화 아키텍처 도식

### 5.2 제품 전체 구조

도식:

```text
입력
ERP / 웹메일 / 웹하드 / 엑셀 / 사진 / 기존 문서

처리
데이터 수집 / 분류 / AI 문서 생성 / 규칙 엔진 / 승인 알림

출력
보고서 / 대장 / 대시보드 / ERP 반영 / 알림
```

### 5.3 제품 모듈

#### 1. ERP / 사내 시스템 연동

설명:

- ERP, DB, 웹하드, 웹메일, 내부 프로그램 연결
- 데이터 자동 수집
- 기존 프로그램을 유지한 상태로 자동화

적용 예:

- 웹메일 첨부파일 자동 분류
- 웹하드 자료 정리
- ERP 입력 자동화
- 승인 요청 자동화

#### 2. 업무 자동화

설명:

- 반복 확인, 분류, 입력, 알림 업무를 자동 처리
- 담당자별 규칙 적용

적용 예:

- 현장 방문 일정 정리
- 파일 업로드 후 자동 알림
- 상태 변경에 따른 담당자 알림
- 누락 업무 체크

#### 3. 문서 자동화

설명:

- 기존 문서 양식에 맞춰 보고서, 대장, 점검표 생성
- 사진, 점검내용, 현장 정보 자동 삽입

적용 예:

- 안전지도 보고서
- 안전보건 대장
- 견적서
- 점검표
- 월간 보고서

#### 4. 대시보드 / 운영 관리

설명:

- 처리 현황과 누락 항목을 관리자 화면에서 확인
- 현장, 날짜, 담당자, 품목 기준으로 관리

적용 예:

- 현장별 보고서 현황
- 안전보건 대장 업데이트 현황
- 재고 부족 알림
- 처리 완료/대기/누락 상태

### 5.4 대표 산업별 패키지

#### 건설 안전/감리 법인

구성:

- 현장 방문 일정
- 사진/점검 데이터 정리
- 안전지도 보고서 생성
- 안전보건 대장 반영
- 관리자 대시보드

#### 제조업

구성:

- 입출고 데이터 정리
- 재고 현황 대시보드
- 부족 재고 알림
- 보고서 자동 생성
- ERP 반영

### 5.5 작동 방식

단계:

```text
1. 기존 업무 확인
2. 데이터 출처 연결
3. 자동화 규칙 설계
4. 문서/대시보드 생성
5. 운영 피드백 반영
```

## 7. MVP 페이지 구조

URL: `/mvp`  
목적: 큰 프로젝트가 부담스러운 고객에게 "작게 시작하는 자동화"를 제안한다.

이 페이지는 대한산업AI에 중요하다. 대부분의 고객은 AI 도입 욕구가 있지만, 처음부터 큰 SI 프로젝트를 결정하기 어렵다. MVP 페이지는 그 진입 장벽을 낮춘다.

### 7.1 MVP Hero

핵심 메시지:

```text
반복 업무 하나부터 MVP로 자동화를 시작합니다.
```

보조 문구:

```text
처음부터 전체 시스템을 바꾸지 않습니다.
가장 반복적이고 효과가 큰 업무 하나를 골라 빠르게 자동화 가능성을 검증합니다.
```

CTA:

- MVP 상담하기
- 자동화 후보 보기

### 7.2 MVP가 필요한 상황

카드:

- AI는 도입하고 싶은데 어디부터 시작할지 모른다.
- 사내 시스템을 한 번에 바꾸기는 부담스럽다.
- 수작업 하나만 먼저 줄여보고 싶다.
- 실제 효과를 보고 본 구축을 결정하고 싶다.

### 7.3 MVP 대상 업무

예시:

- 안전지도 보고서 자동 생성
- 안전보건 대장 자동 업데이트
- 메일 첨부파일 자동 분류
- 웹하드 자료 정리
- 엑셀 → 보고서 자동 생성
- 재고 부족 알림
- 관리자 대시보드 프로토타입

### 7.4 MVP 진행 방식

단계:

```text
1. 업무 후보 선정
2. 샘플 데이터/문서 확인
3. 자동화 흐름 설계
4. MVP 화면 또는 자동화 스크립트 구축
5. 실제 업무에 적용해 검증
6. 본 구축 범위 결정
```

### 7.5 MVP 산출물

가능한 산출물:

- 자동화 데모
- 관리자 화면
- 보고서 자동 생성 샘플
- 대시보드 프로토타입
- 연동 가능성 검토서
- 본 구축 제안 범위

### 7.6 MVP 이후 선택지

```text
MVP 종료
├─ 효과 없음 → 종료 또는 범위 재조정
├─ 일부 효과 있음 → 특정 업무만 운영 자동화
└─ 효과 명확함 → 본 구축 / 시스템 확장
```

### 7.7 MVP CTA

## 13. 디자인 원칙

- 네이비 기반의 고급스럽고 신뢰감 있는 톤
- 문장보다 도식과 제품 화면 중심

- 카드형 UI는 반복 항목에만 사용
- 페이지 섹션은 충분한 여백을 둔다.
- 산업 현장 느낌은 과하게 거칠게 가지 않고, 정돈된 B2B 기술회사 톤을 유지한다.
